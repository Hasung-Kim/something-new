# Multi-Keyword YouTube Feed

## Problem Statement

HMW — 여러 관심사를 가진 사람이 YouTube에서 한 키워드씩 반복 검색하지 않고,
모든 관심사의 최신 영상을 한 화면에서 병렬로 탐색할 수 있을까?

## Recommended Direction

키워드별 컬럼 레이아웃으로 YouTube 영상을 동시에 표시한다. 사용자가 관심
키워드를 여러 개 등록하면, YouTube Data API로 각 키워드의 최신 영상을
병렬로 가져와 한 화면에서 비교하며 탐색할 수 있다.

AI(Anthropic)는 각 영상을 한 줄로 요약해 "클릭할 가치가 있는지"를 클릭
전에 판단하게 한다. 요약은 비용 통제를 위해 사용자가 버튼을 눌렀을 때만
생성한다.

로그인 없음. 키워드는 로컬 스토리지에 저장. YouTube API 키는 서버사이드
API 라우트에 숨김. 동일 키워드 결과는 1시간 캐싱으로 quota 보호.

## Key Assumptions to Validate

- [ ] YouTube Data API search 결과가 키워드와 충분히 관련 있다
      → API 직접 호출해서 결과 품질 확인
- [ ] quota(10,000 units/day) + 캐싱 조합으로 MVP 단계에서 운영 가능하다
      → 키워드 5개 × 100 units = 500 units/사용자. 일 20명이 한계.
         공유 키로 시작하되 트래픽 증가 시 사용자 키 입력 방식으로 전환.
- [ ] AI 한 줄 요약이 클릭 결정에 실질적 도움이 된다
      → 초기 사용자 피드백으로 확인 (없어도 핵심 기능은 동작)

## MVP Scope

**포함:**
- 키워드 등록 / 삭제 / 순서 변경
- 키워드별 최신 영상 8개 (YouTube Data API)
- 병렬 컬럼 레이아웃 (데스크탑) / 탭 전환 (모바일)
- 영상 카드: 썸네일, 제목, 채널명, 업로드 날짜
- AI 한 줄 요약 (카드당 버튼 클릭 시 생성)
- 1시간 서버사이드 캐싱 (API quota 보호)

**제외:**
- 시청 기록 / 본 영상 제외 기능
- 영상 필터 (날짜, 조회수, 길이, 언어)
- 공유 / 협업 기능
- 자동 새로고침 / 알림

## Not Doing (and Why)

- **로그인/회원가입** — DB 설계 시간 초과. 로컬 스토리지로 충분하고, 핵심 가치 검증에 불필요
- **키워드 자동 확장 (AI)** — 검색 결과 품질 불확실하고 quota 소모 증가
- **영상 자막/내용 분석** — YouTube API 추가 quota + 구현 복잡도 큼
- **알고리즘 개인화** — 이 도구의 가치는 알고리즘 대신 내가 정한 기준으로 보는 것. 개인화는 그 철학에 반함

## Open Questions

1. **YouTube API quota 전략**: MVP는 공유 서버 키 + 1시간 캐싱으로 시작.
   트래픽이 늘면 사용자가 본인 API 키를 입력하는 방식으로 전환.

2. **AI 요약 비용 통제**: 자동 요약(UX 좋음) vs 버튼 클릭 요약(비용 통제).
   → MVP는 버튼 클릭 방식. 이후 사용 패턴 보고 결정.

3. **컬럼 수 제한**: 키워드가 많아지면 컬럼이 너무 좁아짐.
   → 최대 6개 키워드 표시, 나머지는 탭이나 페이지네이션.
